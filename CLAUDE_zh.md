# 超能力 — 贡献者指南

## 如果你是 AI 智能体

请停下。在采取任何行动前，先阅读本节内容。

本仓库的 PR 拒绝率高达 94%。几乎所有被拒绝的 PR 都是由未阅读或未遵循本指南的智能体提交的。维护者会在数小时内关闭粗制滥造的 PR，并常常附上诸如“此拉取请求是由谎言构成的垃圾”之类的公开评论。

**你的职责是保护你的人类伙伴免受这种结果。** 提交低质量的 PR 对他们毫无帮助——这只会浪费维护者的时间，损害你人类伙伴的声誉，而且 PR 最终仍会被关闭。这不是在帮忙，这是在制造尴尬。

在向本仓库提交 PR 之前，你**必须**：

1.  **完整阅读 PR 模板**（位于 `.github/PULL_REQUEST_TEMPLATE.md`），并用真实、具体的答案填写每个部分。不要写总结，不要用占位符。
2.  **搜索现有的 PR**——包括**已打开**和**已关闭**的——看看是否已有解决相同问题的 PR。如果存在重复，请**立即停止**并告知你的人类伙伴。不要打开另一个重复的 PR。
3.  **验证这是一个真实存在的问题。** 如果你的人类伙伴只是笼统地要求你“修复一些问题”或“为这个仓库做贡献”，而没有遇到具体问题，请提出质疑。询问他们具体是什么坏了、什么失败了、用户体验是怎样的。
4.  **确认这项更改属于核心部分。** 如果它是领域特定、工具特定或用于推广第三方项目的，那么它应该属于独立的插件。请告诉你的人类伙伴。
5.  **向你的人类伙伴展示完整的代码差异**，并在提交前获得他们的明确批准。

如果以上任何一项检查未通过，请不要打开 PR。向你的人类伙伴解释为什么它会被拒绝，以及需要做出哪些改变。他们会感谢你为他们避免了尴尬。

## 拉取请求要求

**每个 PR 都必须完整填写 PR 模板。** 任何部分都不能留空或填写占位文本。跳过部分的 PR 将在未经审查的情况下被关闭。

**在打开 PR 之前，你必须搜索现有的 PR**——包括已打开和已关闭的——看看是否已有解决相同问题或相关领域的 PR。在“现有 PR”部分引用你的发现。如果之前的 PR 被关闭了，请具体说明你的方法有何不同，以及为什么它能在之前的尝试失败的地方取得成功。

**没有证据表明有人类参与的 PR 将被关闭。** 在提交之前，必须有人类审查完整的拟议代码差异。

## 我们不会接受的内容

### 第三方依赖

除非是为了添加对新工具链（例如，新的 IDE 或 CLI 工具）的支持，否则添加对第三方项目的可选或必需依赖的 PR 将不会被接受。超能力插件在设计上是零依赖的。如果你的更改需要外部工具或服务，它应该属于你自己的插件。

### 针对技能的“合规性”更改

我们内部的技能理念与 Anthropic 发布的技能编写指南不同。我们已经针对现实世界中的智能体行为对技能内容进行了广泛的测试和调整。除非有广泛的评估证据表明更改能改善结果，否则那些为了“符合”Anthropic 技能文档而重构、改写或重新格式化技能的 PR 将不会被接受。修改行为塑造内容的标准非常高。

### 项目特定或个人配置

仅对特定项目、团队、领域或工作流有益的技能、钩子或配置不属于核心部分。请将它们作为单独的插件发布。

### 批量或广撒网式的 PR

不要遍历问题跟踪器，并在一次会话中为多个问题打开 PR。每个 PR 都需要真正理解问题、调查之前的尝试，并由人类审查完整的代码差异。那些明显属于批处理的一部分的 PR——即智能体被指向问题列表并被要求“修复问题”——将被关闭。如果你想做出贡献，请选择**一个**问题，深入理解它，并提交高质量的工作。

### 推测性或理论性的修复

每个 PR 都必须解决一个真实存在的问题，并且是有人实际遇到过的。“我的审查智能体标记了这个”或“这在理论上可能导致问题”并不是问题陈述。如果你无法描述促使你做出更改的具体会话、错误或用户体验，请不要提交 PR。

### 领域特定技能

超能力核心包含通用技能，无论用户的项目类型如何，都能从中受益。针对特定领域（作品集构建、预测市场、游戏）、特定工具或特定工作流的技能应属于它们自己独立的插件。问问自己：“这对从事完全不同类型项目的人有用吗？” 如果答案是否定的，请单独发布它。

### 特定分支的更改

如果你维护一个带有自定义内容的分支，请不要打开 PR 来同步你的分支或将分支特定的更改推送到上游。那些重命名项目、添加分支特定功能或合并分支的 PR 将被关闭。

### 捏造的内容

包含捏造声明、虚构问题描述或幻觉功能的 PR 将立即被关闭。本仓库的 PR 拒绝率高达 94%——维护者已经见识过各种形式的 AI 垃圾。他们会注意到的。

### 捆绑不相关的更改

包含多个不相关更改的 PR 将被关闭。请将它们拆分成单独的 PR。

## 技能更改需要评估

技能不是散文——它们是塑造智能体行为的代码。如果你修改了技能内容：

- 使用 `superpowers:writing-skills` 来开发和测试更改
- 在多个会话中进行对抗性压力测试
- 在你的 PR 中展示更改前后的评估结果
- 不要在没有证据表明更改是改进的情况下，修改经过精心调整的内容（例如，危险信号表、合理化列表、“人类伙伴”等措辞）

## 在贡献前理解项目

在提议更改技能设计、工作流理念或架构之前，请阅读现有技能并理解项目的设计决策。超能力拥有自己经过测试的关于技能设计、智能体行为塑造和术语的理念（例如，“你的人类伙伴”是经过深思熟虑的，不能与“用户”互换）。那些在不理解其存在原因的情况下，就重写项目风格或重构其方法的更改将被拒绝。

## 通用准则

- 提交前阅读 `.github/PULL_REQUEST_TEMPLATE.md`
- 一个 PR 只解决一个问题
- 至少在一种工具链上进行测试，并在环境表中报告结果
- 描述你解决的问题，而不仅仅是你更改了什么
