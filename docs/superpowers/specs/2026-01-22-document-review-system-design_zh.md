# 文档评审系统设计

## 概述

在超级工作流中新增两个评审阶段：

1. **规格文档评审** - 头脑风暴之后，撰写计划之前
2. **计划文档评审** - 撰写计划之后，实施之前

两者均遵循实施评审所使用的迭代循环模式。

## 规格文档评审员

**目的：** 验证规格文档是否完整、一致，并已准备好进行实施规划。

**位置：** `skills/brainstorming/spec-document-reviewer-prompt.md`

**检查内容：**

| 类别 | 检查要点 |
|------|----------|
| 完整性 | TODOs、占位符、"TBD"、不完整的章节 |
| 覆盖度 | 缺失的错误处理、边界情况、集成点 |
| 一致性 | 内部矛盾、冲突的需求 |
| 清晰度 | 模糊的需求 |
| YAGNI | 未要求的功能、过度设计 |

**输出格式：**
```
## 规格评审

**状态：** 已批准 | 发现问题

**问题（如有）：**
- [章节 X]: [问题] - [重要性说明]

**建议（咨询性）：**
- [不阻碍批准的建议]
```

**评审循环：** 发现问题 -> 头脑风暴代理修复 -> 重新评审 -> 重复直至批准。

**调度机制：** 使用 Task 工具，设置 `subagent_type: general-purpose`。评审员提示模板提供完整提示。头脑风暴技能的控制器负责调度评审员。

## 计划文档评审员

**目的：** 验证计划是否完整、符合规格，并具有适当的任务分解。

**位置：** `skills/writing-plans/plan-document-reviewer-prompt.md`

**检查内容：**

| 类别 | 检查要点 |
|------|----------|
| 完整性 | TODOs、占位符、不完整的任务 |
| 规格对齐 | 计划覆盖规格需求，无范围蔓延 |
| 任务分解 | 任务原子化，边界清晰 |
| 任务语法 | 任务和步骤的复选框语法 |
| 块大小 | 每个块不超过 1000 行 |

**块定义：** 块是计划文档内任务的逻辑分组，由 `## 块 N: <名称>` 标题分隔。撰写计划技能根据逻辑阶段（例如"基础"、"核心功能"、"集成"）创建这些边界。每个块应足够自包含，以便独立评审。

**规格对齐验证：** 评审员接收以下两项：
1. 计划文档（或当前块）
2. 规格文档的路径（供参考）

评审员阅读两者并比较需求覆盖情况。

**输出格式：** 与规格评审员相同，但范围限定在当前块。

**评审流程（逐块进行）：**
1. 撰写计划创建块 N
2. 控制器调度计划文档评审员，传入块 N 内容和规格路径
3. 评审员阅读块和规格，返回裁决
4. 若发现问题：撰写计划代理修复块 N，转到步骤 2
5. 若批准：继续处理块 N+1
6. 重复直至所有块批准

**调度机制：** 与规格评审员相同 - 使用 Task 工具，设置 `subagent_type: general-purpose`。

## 更新后的工作流

```
头脑风暴 -> 规格 -> 规格评审循环 -> 撰写计划 -> 计划 -> 计划评审循环 -> 实施
```

**规格评审循环：**
1. 规格完成
2. 调度评审员
3. 若发现问题：修复 -> 转到 2
4. 若批准：继续

**计划评审循环：**
1. 块 N 完成
2. 为块 N 调度评审员
3. 若发现问题：修复 -> 转到 2
4. 若批准：处理下一块或进入实施

## Markdown 任务语法

任务和步骤使用复选框语法：

```markdown
- [ ] ### 任务 1: 名称

- [ ] **步骤 1:** 描述
  - 文件: 路径
  - 命令: cmd
```

## 错误处理

**评审循环终止：**
- 无硬性迭代限制 - 循环持续直至评审员批准
- 若循环超过 5 次迭代，控制器应将此情况上报给人类以获取指导
- 人类可以选择：继续迭代、批准（已知问题）或中止

**分歧处理：**
- 评审员是咨询性的 - 他们标记问题但不阻止
- 若代理认为评审员反馈不正确，应在修复中解释原因
- 若同一问题在 3 次迭代后仍存在分歧，上报给人类

**评审员输出格式错误：**
- 控制器应验证评审员输出是否包含必填字段（状态、问题（如适用））
- 若格式错误，重新调度评审员并附上关于预期格式的说明
- 2 次格式错误响应后，上报给人类
