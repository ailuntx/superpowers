# 测试反模式

**何时参考本文档：** 编写或修改测试时、添加模拟对象时、或想在生产代码中添加仅供测试使用的方法时。

## 概述

测试必须验证真实行为，而非模拟行为。模拟是一种隔离手段，而非被测试对象。

**核心原则：** 测试代码的实际功能，而非模拟对象的行为。

**严格遵循TDD可避免这些反模式。**

## 铁律

```
1. 绝不测试模拟对象的行为
2. 绝不向生产类添加仅供测试使用的方法
3. 绝不未理解依赖关系就进行模拟
```

## 反模式1：测试模拟行为

**违规示例：**
```typescript
// ❌ 错误：测试模拟对象是否存在
test('渲染侧边栏', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**错误原因：**
- 验证的是模拟对象是否工作，而非组件是否工作
- 模拟对象存在时测试通过，不存在时测试失败
- 无法反映真实行为

**人类搭档的纠正：** "我们是在测试模拟对象的行为吗？"

**修正方案：**
```typescript
// ✅ 正确：测试真实组件或不进行模拟
test('渲染侧边栏', () => {
  render(<Page />);  // 不模拟侧边栏
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});

// 或者如果必须为隔离而模拟侧边栏：
// 不对模拟对象进行断言 - 测试侧边栏存在时页面的行为
```

### 检查函数

```
在对任何模拟元素进行断言前：
  自问："我是在测试真实组件行为还是仅仅验证模拟对象存在？"

  如果是测试模拟对象存在：
    停止 - 删除该断言或取消组件模拟

  改为测试真实行为
```

## 反模式2：生产代码中的测试专用方法

**违规示例：**
```typescript
// ❌ 错误：destroy()方法仅在测试中使用
class Session {
  async destroy() {  // 看起来像生产API！
    await this._workspaceManager?.destroyWorkspace(this.id);
    // ... 清理操作
  }
}

// 在测试中
afterEach(() => session.destroy());
```

**错误原因：**
- 生产类被测试专用代码污染
- 生产环境意外调用可能导致危险
- 违反YAGNI原则和关注点分离
- 混淆对象生命周期与实体生命周期

**修正方案：**
```typescript
// ✅ 正确：测试工具处理测试清理
// Session没有destroy()方法 - 在生产环境中它是无状态的

// 在test-utils/中
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}

// 在测试中
afterEach(() => cleanupSession(session));
```

### 检查函数

```
在向生产类添加任何方法前：
  自问："这个方法是否仅供测试使用？"

  如果是：
    停止 - 不要添加
    将其放入测试工具中

  自问："这个类是否拥有该资源的生命周期管理权？"

  如果不是：
    停止 - 这个方法属于错误的类
```

## 反模式3：未理解依赖就进行模拟

**违规示例：**
```typescript
// ❌ 错误：模拟破坏了测试逻辑
test('检测重复服务器', () => {
  // 模拟阻止了测试依赖的配置写入！
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // 应该抛出错误 - 但不会！
});
```

**错误原因：**
- 被模拟的方法具有测试依赖的副作用（写入配置）
- 过度模拟以"确保安全"破坏了实际行为
- 测试因错误原因通过或神秘失败

**修正方案：**
```typescript
// ✅ 正确：在正确层级进行模拟
test('检测重复服务器', () => {
  // 模拟缓慢部分，保留测试所需行为
  vi.mock('MCPServerManager'); // 仅模拟缓慢的服务器启动

  await addServer(config);  // 配置被写入
  await addServer(config);  // 重复检测成功 ✓
});
```

### 检查函数

```
在模拟任何方法前：
  停止 - 先不要模拟

  1. 自问："真实方法有哪些副作用？"
  2. 自问："这个测试是否依赖任何这些副作用？"
  3. 自问："我是否完全理解这个测试需要什么？"

  如果依赖副作用：
    在更低层级模拟（实际的缓慢/外部操作）
    或使用保留必要行为的测试替身
    而不是模拟测试依赖的高层级方法

  如果不确定测试依赖什么：
    首先使用真实实现运行测试
    观察实际需要发生什么
    然后在正确层级添加最小化的模拟

  危险信号：
    - "我模拟这个以确保安全"
    - "这可能很慢，最好模拟它"
    - 未理解依赖链就进行模拟
```

## 反模式4：不完整的模拟

**违规示例：**
```typescript
// ❌ 错误：部分模拟 - 只包含你认为需要的字段
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' }
  // 缺失：下游代码使用的元数据
};

// 后续：当代码访问response.metadata.requestId时会中断
```

**错误原因：**
- **部分模拟隐藏了结构假设** - 你只模拟了你知道的字段
- **下游代码可能依赖你未包含的字段** - 静默失败
- **测试通过但集成失败** - 模拟不完整，真实API完整
- **虚假信心** - 测试无法证明任何真实行为

**铁律：** 模拟现实中存在的完整数据结构，而不仅仅是你当前测试使用的字段。

**修正方案：**
```typescript
// ✅ 正确：反映真实API的完整性
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' },
  metadata: { requestId: 'req-789', timestamp: 1234567890 }
  // 包含真实API返回的所有字段
};
```

### 检查函数

```
在创建模拟响应前：
  检查："真实API响应包含哪些字段？"

  操作：
    1. 从文档/示例检查实际API响应
    2. 包含系统下游可能使用的所有字段
    3. 验证模拟完全匹配真实响应模式

  关键：
    如果你创建模拟，必须理解完整结构
    当代码依赖被省略的字段时，部分模拟会静默失败

  如果不确定：包含所有文档化的字段
```

## 反模式5：集成测试作为事后补充

**违规示例：**
```
✅ 实现完成
❌ 未编写测试
"准备测试"
```

**错误原因：**
- 测试是实施的一部分，不是可选的后续工作
- TDD本应发现这个问题
- 没有测试不能声称完成

**修正方案：**
```
TDD循环：
1. 编写失败测试
2. 实施以通过测试
3. 重构
4. 然后声称完成
```

## 当模拟变得过于复杂时

**警告信号：**
- 模拟设置比测试逻辑更长
- 模拟所有内容以使测试通过
- 模拟缺少真实组件具有的方法
- 模拟更改时测试中断

**人类搭档的问题：** "我们真的需要在这里使用模拟吗？"

**考虑：** 使用真实组件的集成测试通常比复杂模拟更简单

## TDD防止这些反模式

**TDD为何有帮助：**
1. **先写测试** → 迫使你思考实际要测试什么
2. **观察失败** → 确认测试的是真实行为而非模拟
3. **最小化实现** → 防止测试专用方法渗入
4. **真实依赖** → 在模拟前看到测试实际需要什么

**如果你在测试模拟行为，就违反了TDD** - 你在未观察测试对真实代码失败前就添加了模拟。

## 快速参考

| 反模式 | 修正方案 |
|--------|----------|
| 对模拟元素进行断言 | 测试真实组件或取消模拟 |
| 生产代码中的测试专用方法 | 移至测试工具 |
| 未理解就进行模拟 | 先理解依赖关系，最小化模拟 |
| 不完整的模拟 | 完全反映真实API |
| 测试作为事后补充 | TDD - 测试先行 |
| 过度复杂的模拟 | 考虑集成测试 |

## 危险信号

- 断言检查`*-mock`测试ID
- 方法仅在测试文件中调用
- 模拟设置占测试的50%以上
- 移除模拟时测试失败
- 无法解释为何需要模拟
- "为确保安全"而模拟

## 核心要点

**模拟是隔离工具，而非测试对象。**

如果TDD揭示你在测试模拟行为，说明你走错了方向。

修正：测试真实行为，或质疑为何需要模拟。
