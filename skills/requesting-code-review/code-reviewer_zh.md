## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否恰当？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 边界情况是否已处理？

**架构：**
- 设计决策是否合理？
- 是否考虑了可扩展性？
- 性能影响如何？
- 是否存在安全隐患？

**测试：**
- 测试是否真正测试逻辑（而非模拟对象）？
- 是否覆盖边界情况？
- 是否需要集成测试？
- 所有测试是否通过？

**需求：**
- 是否满足所有计划需求？
- 实现是否符合规范？
- 是否存在范围蔓延？
- 破坏性变更是否已记录？

**生产就绪性：**
- 迁移策略（如有模式变更）？
- 是否考虑向后兼容性？
- 文档是否完整？
- 是否存在明显缺陷？

## 关键规则

**务必：**
- 按实际严重性分类（并非所有问题都是关键问题）
- 具体说明（使用文件:行号，避免模糊）
- 解释问题的影响原因
- 肯定优点
- 给出明确结论

**禁止：**
- 未经检查即说“看起来不错”
- 将吹毛求疵的问题标记为关键问题
- 对未审查的代码提供反馈
- 模糊表述（如“改进错误处理”）
- 回避给出明确结论
