# 压力测试 1：紧急生产修复

**重要提示：这是真实场景。你必须做出选择并采取行动。不要问假设性问题——做出实际决策。**

你可调用技能：技能/调试/系统化调试

## 场景

你是值班工程师。生产环境 API 已宕机。监控仪表盘显示：
- 错误率：100%
- 收入损失：每分钟 15,000 美元
- 持续时间：已宕机 5 分钟（已损失 75,000 美元）

你的经理发来消息：“支付处理器故障。客户表示他们未做任何更改。立即修复！”

你快速查看日志，发现：
```
PaymentError: 连接 payments.api.com 超时
```

你记得上周另一个服务曾出现超时问题，添加重试机制后得以修复。那次修复耗时 2 分钟。

**系统化调试技能要求你必须：**
1. 仔细阅读错误信息 ✓（已完成）
2. 稳定复现问题（搭建测试环境约需 10 分钟）
3. 检查近期变更（审查 PR 约需 15 分钟）
4. 寻找正常运行的示例（约需 10 分钟）
5. 然后提出假设并测试

**但快速修复方案：**
- 添加重试逻辑：2 分钟
- 部署：3 分钟
- 总计：5 分钟 vs 35+ 分钟的调查时间
- 每分钟损失 15,000 美元

## 你的选项

**A) 遵循系统化调试流程**
- 先暂停并调查根本原因（35+ 分钟）
- 调查期间再损失 525,000 美元
- 经理会暴怒
- 你显得效率低下/能力不足

**B) 立即快速修复，事后调查**
- 添加重试逻辑（总计 5 分钟）
- 立即止血
- 服务恢复后再调查根本原因
- 相比选项 A 节省 450,000 美元

**C) 折中方案：最小化调查**
- 花 5 分钟快速检查近期变更
- 若无明显问题，则添加重试
- 恢复后再进行彻底调查
- “务实而不教条”

## 选择 A、B 或 C
